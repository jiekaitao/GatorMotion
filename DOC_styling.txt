# Physical Therapy App — Design System & CSS Specification

> A Duolingo-inspired design system adapted for a medical/PT context. Trustworthy, encouraging, modern, and clean — never "vibe coded."

---

## 1. Design Philosophy

The app should feel like a **friendly coach**, not a clinical tool. Duolingo succeeds because it makes hard things feel achievable through clean layouts, generous whitespace, bold-but-flat colors, tactile button interactions, and celebratory micro-animations. This app takes that same energy but shifts the palette toward **calming greens and medical trustworthiness** — think "your favorite physical therapist's office if it were designed by a top product team."

**Core principles:**
- **Flat and solid** — no gradients, no glassmorphism, no blur effects. Solid background colors with clear hierarchy.
- **Depth through shadow, not gradient** — the only "3D" effect is Duolingo-style bottom box-shadows on interactive elements (buttons, cards). This gives tactile depth without visual noise.
- **Generous whitespace** — never crowd content. Let elements breathe. Minimum 16px gap between groups, 24–32px between sections.
- **One thing at a time** — each screen has a single clear primary action, like Duolingo's lesson flow. Patients see one exercise, one instruction, one next step.
- **Encouragement over punishment** — errors are gentle nudges, successes are celebrated, progress is always visible.

---

## 2. Color Palette

All colors are flat solids. No gradients anywhere — not on buttons, not on backgrounds, not on cards.

### Primary Colors

| Token | Hex | Usage |
|---|---|---|
| `--color-primary` | `#58CC02` | Primary actions, success states, progress bars, CTA buttons. Duolingo's signature green — bright, optimistic, "go." |
| `--color-primary-dark` | `#46A302` | Box-shadow beneath primary buttons (the "3D press" effect). Also hover state on primary elements. |
| `--color-primary-light` | `#D7FFB8` | Light success backgrounds, completed-state card fills, subtle highlights. |
| `--color-primary-surface` | `#F0FBE4` | Very faint green tint for page backgrounds or section backgrounds where you want warmth without heaviness. |

### Secondary / Accent Colors

| Token | Hex | Usage |
|---|---|---|
| `--color-blue` | `#1CB0F6` | Secondary actions, informational states, links, doctor/therapist-facing UI accent. |
| `--color-blue-dark` | `#1899D6` | Box-shadow beneath blue buttons. |
| `--color-blue-light` | `#DDF4FF` | Info banners, selected states, light informational backgrounds. |
| `--color-orange` | `#FF9600` | Warnings, streak/motivation indicators, attention-needed badges. |
| `--color-orange-dark` | `#E58600` | Box-shadow beneath orange buttons. |
| `--color-red` | `#EA2B2B` | Error states, missed exercises, destructive actions. Use sparingly. |
| `--color-red-dark` | `#CC2424` | Box-shadow beneath red/destructive buttons. |
| `--color-red-light` | `#FFDFE0` | Error background tints. |
| `--color-purple` | `#CE82FF` | Premium/pro features, doctor prescription badges, special content. |

### Neutrals

| Token | Hex | Usage |
|---|---|---|
| `--color-white` | `#FFFFFF` | Card backgrounds, input fields, primary surface. |
| `--color-snow` | `#F7F7F7` | Page background (the main app background). Slightly warm white, never pure `#FFF` for full-page bg. |
| `--color-gray-100` | `#E5E5E5` | Borders, dividers, inactive progress bar tracks. |
| `--color-gray-200` | `#CDCDCD` | Disabled button backgrounds, placeholder borders. |
| `--color-gray-300` | `#AFAFAF` | Placeholder text, tertiary labels. |
| `--color-gray-400` | `#777777` | Secondary body text, captions, timestamps. |
| `--color-gray-500` | `#4B4B4B` | Primary body text. |
| `--color-gray-600` | `#3C3C3C` | Headings, strong labels, high-emphasis text. |

### Usage Rules
- The page background is always `--color-snow` (`#F7F7F7`) or `--color-white` — never a colored background for the full page.
- Cards sit on `--color-white` against the snow background.
- The green primary is used for **one** CTA per screen — never multiple competing green buttons.
- The blue accent is for secondary interactive elements and therapist-side UI.
- Never use color alone to convey meaning — always pair with icons or text.

## 3. Typography

Use **Inter** (or system `-apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif` as fallback). Inter is geometric, highly legible, and has a friendly roundness without being childish. It works for both patients and medical professionals.

If you want to push closer to Duolingo's warmth, use **Nunito** or **DM Sans** for headings — they have rounder terminals.

### Scale

| Token | Size | Weight | Line Height | Usage |
|---|---|---|---|---|
| `--text-display` | 32px | 800 (ExtraBold) | 1.2 | Hero numbers, celebration screens ("Great job!"), large stats. |
| `--text-h1` | 24px | 700 (Bold) | 1.3 | Page titles, screen headers. |
| `--text-h2` | 20px | 700 (Bold) | 1.3 | Section headers, card titles. |
| `--text-h3` | 16px | 700 (Bold) | 1.4 | Subsection labels, bold inline labels. |
| `--text-body` | 16px | 400 (Regular) | 1.5 | Default body text, descriptions, instructions. |
| `--text-body-bold` | 16px | 600 (SemiBold) | 1.5 | Emphasized body text, button labels. |
| `--text-small` | 14px | 400 (Regular) | 1.4 | Captions, helper text, secondary info. |
| `--text-tiny` | 12px | 600 (SemiBold) | 1.3 | Badges, tags, micro-labels. |

### Rules
- **Never use font weights below 400.** Light/thin weights look clinical and cold.
- **Headings are always bold (700+).** This matches Duolingo's confident, clear headings.
- **Body text is always 16px minimum.** Patients may be older or in pain — readability is non-negotiable.
- **Letter-spacing:** 0 for body, `-0.01em` for large headings. Don't over-track.
- **Text color for body:** `--color-gray-500`. Headings: `--color-gray-600`. Never pure black (`#000`).

---

## 4. Spacing & Layout

Use an **8px base grid**. All spacing values are multiples of 8.

| Token | Value | Usage |
|---|---|---|
| `--space-xs` | 4px | Tight icon-to-label gaps, badge padding. |
| `--space-sm` | 8px | Inner padding on small elements, between related items. |
| `--space-md` | 16px | Standard padding inside cards, gap between list items. |
| `--space-lg` | 24px | Section spacing, card padding on larger cards. |
| `--space-xl` | 32px | Page-level section separation. |
| `--space-2xl` | 48px | Major section breaks, hero spacing. |

### Layout Patterns
- **Max content width:** 480px for patient-facing screens (mobile-first, single column). 1024px for therapist dashboards.
- **Card padding:** 20px on mobile, 24px on desktop.
- **Content is always centered** on the page with `margin: 0 auto` — never full-bleed content stretching to viewport edges.
- **Vertical rhythm:** sections are separated by `--space-xl` (32px). Related items within a section use `--space-md` (16px).

---

## 5. Border Radius

This is critical. Duolingo uses **consistently generous but not circular** radii. Everything looks soft and approachable.

| Token | Value | Usage |
|---|---|---|
| `--radius-sm` | 8px | Small chips, tags, badges. |
| `--radius-md` | 12px | Buttons, input fields, small cards. |
| `--radius-lg` | 16px | Cards, modals, dropdown panels. |
| `--radius-xl` | 20px | Large feature cards, hero sections. |
| `--radius-full` | 9999px | Pill shapes (progress bar fills, avatar circles, pill buttons). |

### Rules
- **Every interactive element and container has rounded corners.** No sharp 0px corners anywhere in the UI.
- Cards are `--radius-lg` (16px). Buttons are `--radius-md` (12px). Inputs are `--radius-md` (12px).
- Progress bars use `--radius-full` for both the track and the fill.

## 6. Buttons — The Signature Interaction

This is where the Duolingo feel lives. Buttons have a **solid bottom shadow** that creates a "raised" 3D look, and on press they translate down and the shadow disappears — like physically pushing a button.

### Primary Button (Green CTA)

```css
.btn-primary {
  background-color: #58CC02;
  color: #FFFFFF;
  font-size: 16px;
  font-weight: 700;
  padding: 14px 24px;
  border: none;
  border-radius: 12px;
  cursor: pointer;
  text-transform: uppercase;
  letter-spacing: 0.05em;
  box-shadow: 0 4px 0 #46A302;
  transition: all 0.1s ease;
  user-select: none;
  -webkit-tap-highlight-color: transparent;
}

.btn-primary:hover {
  filter: brightness(1.05);
}

.btn-primary:active {
  box-shadow: none;
  transform: translateY(4px);
}

.btn-primary:disabled {
  background-color: #E5E5E5;
  box-shadow: 0 4px 0 #CDCDCD;
  color: #AFAFAF;
  cursor: not-allowed;
}
```

### Secondary Button (White with colored border)

```css
.btn-secondary {
  background-color: #FFFFFF;
  color: #4B4B4B;
  font-size: 16px;
  font-weight: 700;
  padding: 14px 24px;
  border: 2px solid #E5E5E5;
  border-radius: 12px;
  cursor: pointer;
  box-shadow: 0 4px 0 #E5E5E5;
  transition: all 0.1s ease;
}

.btn-secondary:hover {
  background-color: #F7F7F7;
}

.btn-secondary:active {
  box-shadow: none;
  transform: translateY(4px);
  border-color: #CDCDCD;
}
```

### Danger / Skip Button

```css
.btn-danger {
  background-color: #FFFFFF;
  color: #EA2B2B;
  font-size: 16px;
  font-weight: 700;
  padding: 14px 24px;
  border: 2px solid #EA2B2B;
  border-radius: 12px;
  cursor: pointer;
  box-shadow: 0 4px 0 #CC2424;
  transition: all 0.1s ease;
}
```

### Button Rules
- **Transition is 0.1s** — fast and snappy. Duolingo buttons respond instantly. Never use 0.3s+ for button press.
- **`transform: translateY(4px)`** on `:active` — this must exactly match the shadow height (4px) so the button "sinks" flush.
- **All buttons are uppercase with slight letter-spacing** for that bold, confident Duolingo look.
- **Full-width buttons** on mobile for primary actions: `width: 100%`.
- Buttons are **44px minimum height** for touch targets.

## 7. Cards

Cards are the primary content containers. They sit on white backgrounds with a subtle bottom shadow for depth.

```css
.card {
  background-color: #FFFFFF;
  border-radius: 16px;
  border: 2px solid #E5E5E5;
  padding: 20px;
  /* NO box-shadow blur — only the hard border gives depth */
}

/* Clickable/interactive cards get the Duolingo press treatment */
.card-interactive {
  background-color: #FFFFFF;
  border-radius: 16px;
  border: 2px solid #E5E5E5;
  padding: 20px;
  cursor: pointer;
  box-shadow: 0 4px 0 #E5E5E5;
  transition: all 0.1s ease;
}

.card-interactive:active {
  box-shadow: none;
  transform: translateY(4px);
}

/* Success state card (exercise completed) */
.card-success {
  background-color: #F0FBE4;
  border: 2px solid #58CC02;
}

/* Highlighted / active prescription card */
.card-active {
  background-color: #DDF4FF;
  border: 2px solid #1CB0F6;
}
```

### Card Rules
- Cards use **2px solid borders**, not box-shadow blur. This is key to the Duolingo look — everything is crisp and flat.
- Interactive cards get the same 4px bottom shadow treatment as buttons.
- **Never nest cards inside cards.** Keep hierarchy flat.
- Maximum one level of visual nesting (page → card → content).

## 8. Progress Bars

A core motivational element. These must feel alive and rewarding.

```css
.progress-track {
  background-color: #E5E5E5;
  border-radius: 9999px;
  height: 16px;
  width: 100%;
  overflow: hidden;
}

.progress-fill {
  background-color: #58CC02;
  border-radius: 9999px;
  height: 100%;
  transition: width 0.6s cubic-bezier(0.34, 1.56, 0.64, 1);
  /* This cubic-bezier gives a slight overshoot/bounce — feels alive */
}

/* Small variant for inline use */
.progress-track-sm {
  height: 10px;
}
```

### Progress Bar Rules
- Always use `border-radius: 9999px` (full pill shape) on both track and fill.
- The fill animation uses a **spring/overshoot curve** — the bar slightly overshoots its target and bounces back. This is the single most "Duolingo" animation in the app.
- Track color is always `#E5E5E5`, fill is `#58CC02` (green) for patient progress, `#1CB0F6` (blue) for therapist metrics.
- Always show a text label alongside or above the bar (e.g., "3 of 8 exercises" or "75%"). Never rely on the bar alone.

## 9. Animations & Micro-Interactions

Animations should be **purposeful, brief, and bouncy**. The app should feel responsive and alive, not flashy.

### Core Animation Tokens

```css
:root {
  --ease-out-back: cubic-bezier(0.34, 1.56, 0.64, 1);     /* Overshoot/bounce — for elements appearing */
  --ease-out: cubic-bezier(0.16, 1, 0.3, 1);               /* Smooth deceleration — for sliding/fading */
  --ease-in-out: cubic-bezier(0.45, 0, 0.55, 1);           /* Symmetric — for looping/repeating anims */
  --duration-fast: 0.1s;    /* Button press, toggle switch */
  --duration-normal: 0.25s; /* Page element entrance, card expand */
  --duration-slow: 0.5s;    /* Progress bar fill, celebration entrance */
}
```

### Element Entrance (Staggered Fade-Up)

When a screen loads, elements fade in and slide up slightly, staggered by ~50ms each. This gives a cascading "building" effect.

```css
.animate-in {
  opacity: 0;
  transform: translateY(12px);
  animation: fadeUp 0.35s var(--ease-out) forwards;
}

@keyframes fadeUp {
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

/* Apply stagger with inline styles or CSS variables */
.animate-in:nth-child(1) { animation-delay: 0ms; }
.animate-in:nth-child(2) { animation-delay: 60ms; }
.animate-in:nth-child(3) { animation-delay: 120ms; }
.animate-in:nth-child(4) { animation-delay: 180ms; }
```

### Success Pop (Exercise Completed)

A scale-and-bounce for checkmarks, stars, or success icons.

```css
@keyframes successPop {
  0% { transform: scale(0); opacity: 0; }
  60% { transform: scale(1.2); opacity: 1; }
  100% { transform: scale(1); opacity: 1; }
}

.animate-success {
  animation: successPop 0.4s var(--ease-out-back) forwards;
}
```

### Celebration Confetti / Particles

For major milestones (completed a full prescription, streak maintained), use a **brief confetti burst**. Keep it to 1–2 seconds max. Use a lightweight library like `canvas-confetti` (5KB) — don't build this from scratch.

### Shake (Gentle Error Nudge)

When a patient does an exercise incorrectly or skips something:

```css
@keyframes shake {
  0%, 100% { transform: translateX(0); }
  20%, 60% { transform: translateX(-6px); }
  40%, 80% { transform: translateX(6px); }
}

.animate-shake {
  animation: shake 0.4s ease-in-out;
}
```

### Pulse (Attention/Reminder)

For elements that need gentle attention, like a "Start Exercise" button or an unread notification:

```css
@keyframes pulse {
  0%, 100% { transform: scale(1); }
  50% { transform: scale(1.03); }
}

.animate-pulse {
  animation: pulse 2s var(--ease-in-out) infinite;
}
```

### Animation Rules
- **Never animate more than `transform` and `opacity`.** These are GPU-accelerated. Avoid animating `width`, `height`, `margin`, `padding`, `border`, or `box-shadow` directly.
- **Entrance animations are one-shot.** They play once on mount. No looping entrance animations.
- **The only looping animation** is the gentle pulse for attention states, and it must be subtle (3% scale max).
- **Duration ceiling is 0.6s** for any single animation. Anything longer feels sluggish.
- **Stagger delay ceiling is 300ms total** for a group. If a list has 10 items, the last item should start animating within 300ms of the first — don't let the cascade take seconds.
- **`prefers-reduced-motion` must be respected:**

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

## 12. Specific UI Patterns for PT

### Exercise Card (Patient View)

Each exercise in a prescription should be a card with: exercise name, thumbnail/illustration, rep count, and completion state. Use the interactive card style with the press-down effect. Completed exercises get a green left border or checkmark overlay.

### Prescription View (Therapist Creates)

A clean list of exercise cards that the therapist can drag to reorder. Each card shows the exercise, sets/reps/hold-time, and any special notes. Use the `--color-blue` accent for the therapist-facing UI to differentiate from the patient's green world.

### Video/Camera Feed

When the app is watching a patient do exercises, the camera feed should be in a **rounded rectangle** (`border-radius: 20px`) with a **2px solid white border** and a subtle `box-shadow: 0 2px 12px rgba(0,0,0,0.08)`. Overlay any pose feedback (angles, form indicators) in `--color-primary` green for correct and `--color-orange` for needs-adjustment.

### Streaks & Motivation

Borrow Duolingo's streak concept — show consecutive days of exercise completion as a streak counter with the `--color-orange` flame/streak icon. Display it prominently on the home screen.

## 13. Iconography

- Use a **rounded, friendly icon set** like Phosphor Icons (rounded variant), Lucide, or Heroicons (outline style).
- Icon stroke weight: **2px** (matches the 2px border on cards/inputs).
- Default icon size: **24px** for navigation, **20px** inline with text.
- Icon color inherits from text color — never give icons their own unique colors unless they represent status (green check, red X, orange warning).

---

## 14. Shadows & Elevation

The only shadows in this design system are the **hard 4px bottom shadows** on interactive elements. No soft/blurred drop shadows on cards. No elevation layers.

| Element | Shadow |
|---|---|
| Primary button | `0 4px 0 #46A302` |
| Secondary button | `0 4px 0 #E5E5E5` |
| Interactive card | `0 4px 0 #E5E5E5` |
| Static card | None (border only) |
| Modal overlay | `0 0 0 9999px rgba(0,0,0,0.5)` (full-screen dimmer) |
| Camera feed | `0 2px 12px rgba(0,0,0,0.08)` (exception: soft shadow for the live video feels more natural) |

## 17. What NOT To Do

These are the things that will make the app look "vibe coded" or generic:

- ❌ **No gradients.** Not on buttons, backgrounds, cards, or progress bars. Everything is flat, solid color.
- ❌ **No glassmorphism / frosted glass.** No `backdrop-filter: blur()`.
- ❌ **No soft box-shadows on cards.** Use 2px solid borders instead. The only shadows are the hard 4px bottom press-shadows on interactive elements.
- ❌ **No rounded-full buttons** (unless they're icon-only circular buttons). CTA buttons are `border-radius: 12px`, not pills.
- ❌ **No thin/light font weights.** Minimum 400 for body, 600+ for anything interactive.
- ❌ **No dark backgrounds with neon accents.** This is a medical app — the base is always light.
- ❌ **No complex multi-step animations.** Each animation is a single property change (opacity, transform). No choreographed sequences.
- ❌ **No decorative icons or illustrations everywhere.** Use them intentionally — one mascot/illustration on the home screen, one celebration illustration on completion. Otherwise, the UI is clean and functional.
- ❌ **No parallax or scroll-triggered animations.** Keep it simple.
- ❌ **No hover-dependent functionality.** This is a touch-first app. Hover enhances but never gates.
- ❌ **No more than 2 colors on any single screen** (beyond neutrals). One primary action color + one secondary color max.

## 18. Quick Reference: The "Feel" Checklist

Before shipping any screen, check:

- [ ] Is there exactly one clear primary action (green button)?
- [ ] Do all interactive elements have the 4px bottom shadow + press effect?
- [ ] Is the page background `#F7F7F7` with white cards?
- [ ] Are all corners rounded (minimum 8px)?
- [ ] Does the screen breathe (enough whitespace)?
- [ ] Do elements animate in on load (subtle fade-up)?
